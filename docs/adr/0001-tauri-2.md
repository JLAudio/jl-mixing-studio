# ADR-0001: Use Tauri 2 as the desktop architecture

- **Status:** Accepted
- **Date:** 2026-07-17
- **Decision owners:** JL Mixing Studio maintainers

## Context

JL Mixing Studio needs a cross-platform desktop GUI for macOS and Windows. It must be open source, inexpensive to build and distribute, appropriate for a small contributor community, and capable of safe local filesystem and process integration.

The main candidates considered were:

- Tauri 2 with a web frontend and Rust desktop layer.
- Electron with a web frontend and Node.js desktop layer.
- Native or native-style free frameworks, principally Avalonia and Qt.
- Separate native macOS and Windows implementations.

JL Mixing Studio primarily needs workflow screens, reports, project navigation, JSON metadata handling, filesystem access, and controlled invocation of JL Mixing Automation. It does not need an embedded audio engine or DAW plug-in host.

## Decision

Use the following architecture:

- Tauri 2 for the desktop application shell.
- React and TypeScript for the user interface.
- Rust for application logic that touches the operating system, filesystem, or external processes.
- Typed, narrowly scoped Tauri commands between the frontend and Rust.
- JL Mixing Automation as a versioned external command-line dependency during the initial implementation.

The frontend must not receive unrestricted filesystem or shell access.

## Rationale

Tauri offers the best current balance of:

- Small application footprint by using the operating system WebView.
- Permissive MIT/Apache-2.0 licensing.
- macOS and Windows support.
- A capability-based security model.
- A modern web UI development model.
- Strongly typed native integration in Rust.
- Free development, build, and test tooling.

Electron was not selected because bundling Chromium and Node.js increases installer size, memory use, security-update burden, and pressure to follow Chromium's operating-system support schedule.

Avalonia was not selected because current macOS support tiers place macOS 13 and earlier in Tier 3, affecting the available Intel Monterey development machine. Qt was not selected because LGPL/GPL compliance adds avoidable distribution complexity. Separate native implementations would duplicate too much work.

## Consequences

### Positive

- One cross-platform UI codebase.
- A small and explicit security boundary.
- Strong testability of parsing and application logic.
- Straightforward creation of a polished, studio-oriented interface.
- No framework licensing fees.

### Negative

- Contributors must work with both TypeScript and Rust.
- Rust compilation may be slow on older development hardware.
- System WebView differences require macOS and Windows UI testing.
- Platform packages must be built and validated per operating system.

## Validation gate

The architecture spike validated:

1. Launch on an Intel Mac running macOS Monterey.
2. Build on Windows through reproducible CI.
3. Restricted invocation of `jl-mixing --version`.
4. Typed reading and display of a representative v1.2.0 project manifest.
5. Automated tests and reproducible CI for the supported build environments.

The spike passed its automated macOS and Windows gates and its manual Intel macOS Monterey 12.7.6 validation. Reassess this decision if later platform testing reveals a material incompatibility.

## Guardrails

- No arbitrary shell command API may be exposed to the frontend.
- No direct frontend access to unrestricted filesystem paths.
- No application-only database may become the source of truth for project lifecycle state.
- No paid-only dependency or service without explicit approval.
- No implicit changes to JL Mixing Automation behavior or metadata.
